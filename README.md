# MVP Factory

매주말 MVP 하나를 기획 → 구현 → 검수 → 배포하는 에이전트 하네스 템플릿.
새 빌드는 이 리포를 **Use this template**으로 복제해서 시작한다.

## 철학

- **에이전트 분리**: 기획(lead) / 구현(builder) / 검수(reviewer)를 별도 서브에이전트로 분리. 기획한 에이전트가 자기 결과물을 검수하면 같은 맹점을 두 번 통과하므로, 검수는 깨끗한 컨텍스트가 맡는다.
- **규칙은 훅으로 강제**: 프롬프트는 해석에 의존하지만 훅은 결정론적이다. 포맷팅, 위험 명령 차단, 품질 게이트는 전부 훅이 처리한다.
- **컨텍스트 경제**: CLAUDE.md는 철칙과 포인터만. 상세 규칙은 스킬로 분리해 관련 작업 때만 조건부 로딩. 무거운 구현 로그는 서브에이전트의 격리 컨텍스트에서 소각.

## 5분 시작

```bash
pnpm install
claude   # Claude Code 실행
```

첫 프롬프트: `lead로 이번 주 빌드 PLAN.md 만들어줘. 아이디어: <한 줄>`

## 구조

| 경로              | 역할                                                                           |
| ----------------- | ------------------------------------------------------------------------------ |
| `CLAUDE.md`       | 철칙 5개 + 포인터 (항상 로딩)                                                  |
| `PLAN.md`         | 이번 빌드의 범위 정의 (lead가 생성)                                            |
| `BUILD_LOG.md`    | 의사결정/삽질 로그 — 블로그 글의 원재료                                        |
| `.claude/agents/` | lead(기획) · builder(구현) · reviewer(검수)                                    |
| `.claude/skills/` | architecture · conventions · shipping · log-decision · write-post · blog-style |
| `.claude/hooks/`  | guard(위험 명령 차단) · format(자동 포맷) · verify(종료 전 품질 게이트)        |

## 명령어 & 사용법

호출 방식은 세 가지다. (1) **에이전트** — 자연어로 위임, (2) **스킬** — `/이름` 슬래시 또는 작업 맥락에서 자동 참조, (3) **셸** — `pnpm`.

### 에이전트 (서브에이전트에 위임)

메인 세션은 오케스트레이션만 하고 무거운 작업은 아래로 위임한다.

| 에이전트   | 언제                 | 호출 예시                                                 |
| ---------- | -------------------- | --------------------------------------------------------- |
| `lead`     | 기획 / 범위 변경     | `lead로 이번 주 빌드 PLAN.md 만들어줘. 아이디어: <한 줄>` |
| `builder`  | 구현 / 수정 / 테스트 | `builder로 PLAN.md의 T3 구현해줘`                         |
| `reviewer` | 머지·배포 전 검수    | `reviewer로 지금 변경분 검수해줘` (FAIL이면 배포 불가)    |

### 스킬 (`/이름` 슬래시 명령어)

`/write-post`처럼 직접 부를 수도 있고, 관련 작업을 할 때 에이전트가 알아서 참조하기도 한다.

| 스킬                           | 주 호출                            | 용도                                                            |
| ------------------------------ | ---------------------------------- | --------------------------------------------------------------- |
| `/write-post`                  | 수동 (배포 후)                     | 블로그 글 생성 — 제목 1개 확정 + 본문(H1) + 태그 + 요약         |
| `/shipping`                    | 배포 작업 시                       | 일요일 배포 절차·체크리스트 (CI 더미 env, Supabase 대시보드 등) |
| `/log-decision`                | 결정할 때 (보통 자동)              | BUILD_LOG.md 기록 형식                                          |
| `/architecture` `/conventions` | 코드 작성 전 (builder가 자동 참조) | 폴더 구조·레이어 / 코드 컨벤션                                  |
| `/design`                      | UI 작업 시 (자동 참조)             | 토스 스타일 디자인 시스템                                       |
| `/blog-style`                  | 글쓰기 시 (write-post가 자동 적용) | 블로그 문체                                                     |

### 셸 (`pnpm`)

| 명령                         | 용도                                                                   |
| ---------------------------- | ---------------------------------------------------------------------- |
| `pnpm dev`                   | 개발 서버                                                              |
| `pnpm check`                 | lint + typecheck + test — **이게 통과해야 작업 완료** (Stop 훅이 강제) |
| `pnpm build`                 | 프로덕션 빌드 — 배포 전 로컬에서 빌드 타임 에러 확인                   |
| `pnpm e2e`                   | 스모크 테스트 (있는 경우)                                              |
| `pnpm knip`                  | 미사용 export/파일/의존성 탐지                                         |
| `pnpm post:tistory <md파일>` | 티스토리 반자동 발행 (아래 가이드 참조)                                |

> `git push`·`.env` 읽기·`rm -rf` 등 되돌리기 어려운 명령은 settings/guard 훅이 차단한다. push는 사람이 직접 한다.

### 블로그 포스팅 가이드 (`pnpm post:tistory`)

티스토리 Open API는 종료돼서(2024.2) API 발행은 불가능하다. 그래서 이 스크립트는 **Playwright로 에디터를 자동 입력하고, 최종 [공개 발행] 버튼만 사람이 누르는** 반자동 방식이다.

**0. 글 준비 (Claude Code 안에서)**

1. 배포 완료 + `BUILD_LOG.md`의 Decisions·Stuck & Solved가 채워졌는지 확인 (재료가 없으면 글이 일반론이 된다)
2. `/write-post` 실행 → `posts/<날짜>-<제품>.md`에 완성본 저장 (맨 위 `# 제목` H1 필수 — 스크립트가 이 줄을 제목으로 떼어 쓴다)
3. 스크린샷은 `posts/screenshots/`에 두고 **commit & push** → 본문이 GitHub raw 절대 URL로 임베드돼 티스토리에서 그대로 렌더링된다 (public repo 기준. push 안 하면 이미지가 깨진다)

**1. 최초 1회 셋업**

```bash
pnpm exec playwright install chromium   # 브라우저 바이너리 설치 (한 번만)
```

> 클립보드 폴백(`pbcopy`)은 macOS 기준이다.

**2. 실행**

```bash
pnpm post:tistory posts/2026-06-career-vault.md            # 블로그=mingeonho1, 카테고리=AI (기본값)
pnpm post:tistory posts/2026-06-career-vault.md myblog 회고  # 블로그·카테고리 지정
```

인자: `<md파일경로>` (필수) · `[블로그이름]` (기본 `mingeonho1`) · `[카테고리]` (기본 `AI`)

**3. 실행 후 일어나는 일**

1. 전용 프로필(`~/.tistory-playwright`)로 크롬 창이 열린다 — 로그인 세션은 여기 저장돼 다음부터 재로그인 불필요
2. 로그인 안 돼 있으면 그 창에서 **직접 로그인** (최대 10분 대기, 로그인하면 자동으로 글쓰기 페이지로 진행)
3. 카테고리 선택 → 제목 입력 → 마크다운 모드 전환 → 본문 주입(+길이 검증) → 발행 레이어 열고 '공개' 선택까지 자동
4. **마지막 [공개 발행] 클릭은 사람이** — 내용 확인 후 직접 누른다. 발행하고 브라우저를 닫으면 스크립트 종료

**실패하면:** 본문이 클립보드에 복사돼 있으니 에디터(마크다운 모드)에 `Cmd+V`. 원인은 대개 티스토리 에디터 DOM 변경이고, 같이 저장되는 `tistory-error.png`(또는 `tistory-debug.png`)를 Claude에게 보여주면 셀렉터를 고칠 수 있다.

## 주말 운영 타임라인

| 시간    | 작업                                                                |
| ------- | ------------------------------------------------------------------- |
| 금 저녁 | 아이디어 확정 → lead가 PLAN.md 생성 → Non-goals 승인                |
| 토      | builder 구현 (태스크당 `pnpm check` 통과 필수) → reviewer 1차 검수  |
| 일 오전 | reviewer `SHIP IT` → shipping 스킬 절차로 Vercel 배포 + 계측        |
| 일 오후 | `/write-post`로 글 작성 → `pnpm post:tistory`로 발행 → 아래 표 갱신 |

**킬 규칙**: 배포 2주 내 결제/사용 신호 0이면 유지보수 중단 (리포는 공개 유지).

## 품질 게이트 (3중)

1. 스킬 — 에이전트가 컨벤션/아키텍처 규칙 참조
2. 훅 — Stop 시 `pnpm check` 실패하면 세션 종료 자체가 차단됨
3. CI — push마다 lint + typecheck + test + build + knip
